Claude vs Parallel.ai

Three head-to-head runs on the production external-research stage of the market-research plugin.

24 May 2026 n = 3 companies AEP · WEC · DaVita

Claude won every head-to-head run. Across the three companies tested, the blind LLM judge picked the production Claude path over Parallel.ai's Task API on all five rubric criteria, with the decisive gap on coverage breadth and citation quality.

Parallel.ai is, however, structurally 22× cheaper per run: $0.10 flat per Task API call vs $2.21 mean pay-per-token for Claude. Latency is essentially tied — close enough that the per-company variance swallows the mean difference.

Recommendation

Keep Claude in production for deep-dossier client work — the current use case.

Park Parallel.ai for future batch screening only. Don't build the batch workflow until there's demonstrated demand.

The two cost profiles are structurally different. Claude is metered per token (input, output, cache write, cache read), so cost scales with the depth and length of the agent's work — DaVita finished cheaper than AEP because the sub-agent ran fewer turns. Parallel's pro processor is a flat $0.10 per Task API call regardless of how much research happens under the hood, with no per-token surcharges; the price is the price.

Latency carries a per-run nuance worth surfacing. The mean tilts to Claude by 1.27× (385.68s vs 488.14s), but that headline hides the within-run variance. Parallel actually beat Claude on AEP — 445.9s vs 465.9s, a ~20-second edge to Parallel. The mean only tilts to Claude because WEC and DaVita went the other way by much larger margins (Claude faster by ~200s on WEC and ~127s on DaVita). On a single-company decision, latency is essentially a coin flip; on a batch, Claude has a small edge.

Caveats

Where this report could be wrong

These are integrity content — the difference between a report and a sales pitch. Read them.

  1. n = 3 is a small sample.

    Two regulated electric utilities (AEP, WEC) and one healthcare-services business (DaVita). The pattern observed here may not generalize to consumer-goods, fintech, deep-tech, or any company where the underlying source ecosystem differs. The directional verdict is consistent across all three runs, which is encouraging, but a fourth sector could plausibly invert one criterion.

  2. The blind judge had two provenance leaks during the recorded runs.

    The judge prompt rendered the de-anonymisation mapping into the user prompt, and the JSON outputs visible to the judge still carried researcher and generated_at fields. Both were fixed in commit a4e6ee7, but that commit landed after the runs completed. The directional verdict survives because the judge's findings are independently verifiable (Bloom Energy, FY2024 $19.72B, Berkshire 45% cap, etc. are real facts in Claude's outputs that are absent in Parallel's) and the coverage_breadth gap is too large to explain by bias alone — but the per-criterion magnitudes may be inflated, particularly on the close criteria (recency, structural fidelity).

  3. Claude token attribution is incomplete.

    The claude -p --output-format json response surfaces parent-session tokens only, not the tokens consumed by the spawned external-researcher sub-agent. The total_cost_usd field is accurate — that's the load-bearing number for the cost comparison and the figure used throughout this report — but a per-agent breakdown is not available from the current CLI shape. If finer attribution becomes important, the workaround is parsing the session log.

  4. Two real citation issues surfaced in production Claude output on DaVita.

    First, the 2014 DOJ $350M settlement page was cited as evidence for a claim about the 2024 $34.5M settlement — a time-mismatched citation. Second, Wikipedia was cited for the employee count when the underlying 10-K was already in Claude's citation list — an aggregator chosen over an available primary source. Both are addressed by a citation-discipline patch landed on main (commit 9a080ae); future runs should not exhibit either failure mode.

Technical appendix

Methodology and reproducibility

Claude path

The production external-researcher sub-agent invoked via claude -p, scoped to the external stage only. Entry point: the /market-research:bench-external <company> slash command, which dispatches the sub-agent with explicit output paths into bench/results/claude/<slug>/. Sub-agent tools: WebSearch, WebFetch, Read, Write, Bash. Permission mode: --dangerously-skip-permissions for non-interactive operation.

Parallel.ai path

Task API in Deep Research mode via the parallel-web Python SDK. Processor: pro (blocking; 2–10 min latency band per Parallel docs). output_schema set to the same canonical bench/schema.json the Claude path targets, so both implementations write into an identical shape. Per-field reasoning and citations available via result.output.basis, persisted to parallel_basis.json per run.

Judge configuration

Claude Opus 4.7 via the Anthropic SDK, scoring each pair blind (A vs B random assignment, mapping logged for audit). Rubric: factual_accuracy, citation_quality, coverage_breadth, recency, structural_fidelity — all 1–5 with required reasoning. URL-pattern audit independently classifies a sample of each output's citations and reports over-claims. Mapping is overwritten by the harness post-call, so the judge's own mapping output cannot bias attribution.

Cost calculation

Claude: total_cost_usd from the claude -p --output-format json response, sourced directly from Claude Code's per-model metering. Parallel: $0.10 flat per task per bench/pricing.json (the pro processor's published rate, 100 task-runs per $10). No prompt-cache complications on either side — Claude figures include cache reads/writes naturally; Parallel is per-task with no per-token meter.
